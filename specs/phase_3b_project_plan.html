3B.0
Foundation & prerequisites
1–2 days · no dependencies
  • Add statsforecast, hierarchicalforecast, polars, duckdb, prophet, openpyxl, pandas to requirements.txt
  • Add ClientFeatureControl keys: demand_planning, actuals_upload, forecast_run, consensus_override, forecast_approval
  • Add ClientLocation.parent self-FK + path CharField + index (text_pattern_ops)
  • makemigrations → migrate → manage.py check
Deliverable: Clean migrations; all packages installed; location tree fields present
3B.1
Sales hierarchy models
2–3 days · needs 3B.0
  • SalesNode(client, name, level_label, parent→self, location FK nullable, path) + pre_save signal for materialized path
  • CustomerSalesAssignment(customer, sales_node, valid_from, valid_to)
  • ix_salesnode_path index via RunSQL
  • Admin: tree display with indent; CustomerSalesAssignmentInline
  • REST: GET /api/demand/sales-hierarchy/ and /location-hierarchy/
  • Tests: path on create/reparent; subtree query; date effectivity
Deliverable: Full sales org tree configurable in Admin; hierarchy APIs return correct nested JSON
3B.2
Actuals models & import pipeline
3–4 days · needs 3B.1
  • ActualSale(client, item, variant, customer, location, year, month, qty, revenue) + unique constraint + indexes
  • ActualSaleLocation (summary-level); ActualSaleImport (job tracking)
  • Celery task process_actuals_import: parse CSV/Excel, FK validation, bulk_create with update_conflicts=True
  • REST: POST /actuals/upload/, GET /actuals/upload/{id}/ (poll), GET /actuals/ (filtered query)
  • generate_actuals_template management command → .xlsx download
  • Tests: idempotent re-upload; bad FK → error row only; row count correct
Deliverable: Client staff can upload monthly SKU × Customer × Location actuals; import is idempotent
3B.3
Forecast models & version management
3–4 days · needs 3B.2
  • ForecastVersion(client, version_label, base_period_end, horizon_months, engine_config JSONField, status, created_by, approved_by, approved_at)
  • ForecastLine(version, item, customer, location, year, month, statistical_qty, override_qty, final_qty) — final_qty computed on save
  • ForecastAggregate, ForecastOverride, OverrideSplitWeight, ForecastAccuracy
  • Status machine: DRAFT → IN_REVIEW → APPROVED → LOCKED
  • REST: CRUD on /forecast-versions/, /lines/, /aggregates/, /approve/
  • Tests: final_qty logic; LOCKED rejects edits; version copy
Deliverable: Forecast version lifecycle works; REST endpoints stubbed for SPA development to begin
3B.4
Statistical forecast engine
5–7 days · needs 3B.3 · parallel with 3B.5
  • utils/demand/forecast_engine.py: build_actuals_dataframe (DuckDB → Polars), build_summing_matrix S + tags for 3 hierarchies
  • run_statsforecast: AutoETS, AutoARIMA, CrostonSBA (auto-selected for intermittent series > X% zero months), Theta
  • run_hierarchical_reconciliation: MinTrace OLS/WLS, BottomUp, TopDown AHP — configurable per client
  • write_forecast_lines (bulk_create batched 5000) + write_forecast_aggregates
  • Celery task run_forecast(version_id): orchestrates pipeline, updates version status, notifies user
  • Tests: S matrix shape; reconciled top-level = bottom-up sum; CrostonSBA selection; bulk_create row count
Deliverable: Triggering a ForecastVersion run produces auditable ForecastLine records with reconciled quantities
3B.5
Consensus override engine
4–5 days · needs 3B.3 · parallel with 3B.4
  • utils/demand/override_engine.py: resolve_leaf_lines (subtree walk via materialized path), compute_proportional_shares (from ActualSale history)
  • apply_override → updates ForecastLine.override_qty per leaf
  • recompute_aggregates for affected agg_keys
  • Routing: < 500 leaves → synchronous; ≥ 500 → Celery task propagate_override(override_id)
  • REST: POST/DELETE /overrides/ — delete reverts leaf lines to statistical_qty
  • Tests: disagg sums to total; revert restores statistical; LOCKED rejects POST
Deliverable: Planner can override at any hierarchy level; disaggregation reaches leaf ForecastLines correctly
3B.6
Accuracy tracking
2–3 days · needs 3B.4 + 3B.5
  • Celery task compute_forecast_accuracy(version_id, year, month): join ForecastLine (approved) with ActualSale; compute MAPE, Bias, WMAPE per leaf and aggregate
  • Write ForecastAccuracy records (bulk_create, update_conflicts=True)
  • Monthly Celery beat schedule triggers for all clients with demand_planning enabled
  • REST: GET /forecast-versions/{id}/accuracy/
  • Tests: MAPE formula; WMAPE aggregate; division-by-zero guard (actual=0)
Deliverable: After actuals close, accuracy metrics are computed automatically and accessible via API
3B.7
React SPA frontend
7–10 days · can start after 3B.3 stubs · parallel with 3B.4+3B.5
  • Vite + TypeScript scaffold under frontend/planning/; served at /{client_id}/planning/ via Django catch-all
  • Auth: Django session cookie; DRF SessionAuthentication on all demand endpoints
  • View 1 — Actuals Dashboard: drag-drop upload widget + poll; ECharts Location×Month heatmap; AG Grid SKU drill-down
  • View 2 — Forecast Run: period/horizon/method selectors; submit → poll status; progress bar
  • View 3 — Consensus Grid: AG Grid editable cells; colour coding (white/yellow/grey); POST override on edit
  • View 4 — Version Comparison: two-version ECharts overlay at any aggregate level
  • View 5 — Accuracy Report: MAPE/Bias/WMAPE AG Grid + worst-SKUs bar chart
  • Vitest: cell edit → correct API call; polling logic; share display
Deliverable: Planners can upload, run forecasts, override, compare versions, and view accuracy — all in the browser
3B.8
Phase 3A integration
2–3 days · needs 3B.7 + Phase 3A Sprint 3.3
  • OrderLine.forecast_line nullable FK → ForecastLine
  • On Quotation/Order creation: query approved ForecastVersion for Customer × Item → show suggested qty as hint on line form
  • ForecastAccuracy extended: actual_order_qty column populated from OrderLine quantities in addition to ActualSale uploads
  • Tests: suggested qty shown when approved version exists; FK set on acceptance
Deliverable: Approved forecasts surface as suggested order quantities; plan-to-order loop closed
3B.9
Production hardening
2–3 days · needs all prior sprints
  • Deploy Celery worker with CELERYD_TASK_SOFT_TIME_LIMIT=1800 (forecast jobs up to 30 min)
  • Verify DuckDB pure-Python wheel installs cleanly in production container
  • EXPLAIN ANALYZE on actuals matrix pull; confirm indexes used
  • Load test: 10K SKU × 200 customer × 36 month run < 15 minutes
  • Load test: 50K row actuals upload < 60 seconds
  • E2E: upload → run → override → approve → order suggestion visible
  • Wire ClientGroupPermission (demand_planning module) to all demand API endpoints
Deliverable: Phase 3B live in production; load tests passed; permissions enforced
