# Setting up `gettext_lazy` / `_("...")` Project-Wide
## Django i18n for UI Strings — Complete Setup Instructions

**Scope:** This covers only `gettext_lazy` for code-level UI strings (field labels,
verbose names, error messages, help text, form labels, admin headings).
`django-modeltranslation` remains untouched — it handles DB-stored content fields.

---

## 1. What `_("...")` Actually Does

```python
from django.utils.translation import gettext_lazy as _

verbose_name = _("actual sale")        # lazy: evaluated when rendered
verbose_name = "actual sale"           # eager: evaluated at import time
```

At runtime, `_("actual sale")` looks up `"actual sale"` in the active language's
`.po` file and returns the translation. If no translation exists, it returns the
original string unchanged. So adding `_("...")` is **safe and non-breaking** even
before any `.po` files exist — the strings just pass through as-is.

**lazy vs non-lazy:**
- Use `gettext_lazy as _`  → in models, forms, serializers (module-level code)
- Use `gettext as _`       → inside functions/views (already in request context)
- In practice, `gettext_lazy` works safely in both places — prefer it everywhere

## 2. Django Settings Changes

Open `settings.py` and verify / add these entries:

```python
# ── Internationalisation ──────────────────────────────────────────────────────

USE_I18N = True          # must be True (likely already set)
USE_L10N = True          # number/date localisation (likely already set)

# Languages you support. The first entry is the default.
LANGUAGE_CODE = 'en'     # your default language

LANGUAGES = [
    ('en', 'English'),
    ('hi', 'Hindi'),       # add / remove as needed
    # ('mr', 'Marathi'),
    # ('ta', 'Tamil'),
]

# Where Django looks for .po files that are NOT inside an app
# (project-level translations, e.g. for templates)
LOCALE_PATHS = [
    BASE_DIR / 'locale',
]

MIDDLEWARE = [
    # ... your existing middleware ...
    'django.middleware.locale.LocaleMiddleware',   # ← add this
    # Must be AFTER SessionMiddleware and BEFORE CommonMiddleware:
    # 'django.contrib.sessions.middleware.SessionMiddleware',
    # 'django.middleware.locale.LocaleMiddleware',      ← here
    # 'django.middleware.common.CommonMiddleware',
]
```

**`LocaleMiddleware` position matters.** The correct order is:

```python
MIDDLEWARE = [
    'django.middleware.security.SecurityMiddleware',
    'django.contrib.sessions.middleware.SessionMiddleware',
    'django.middleware.locale.LocaleMiddleware',      # ← after Session
    'django.middleware.common.CommonMiddleware',
    'django.middleware.csrf.CsrfViewMiddleware',
    'django.contrib.auth.middleware.AuthenticationMiddleware',
    'django.contrib.messages.middleware.MessageMiddleware',
    'django.middleware.clickjacking.XFrameOptionsMiddleware',
    # ... allauth, your custom middleware ...
]
```

## 3. Directory Structure for `.po` Files

Django expects translation files at:

```
<app>/locale/<language_code>/LC_MESSAGES/django.po
```

For project-level strings (templates, non-app code):

```
locale/<language_code>/LC_MESSAGES/django.po
```

Create the directories now (files are generated by `makemessages`):

```bash
# Project-level locale dir
mkdir -p locale/en/LC_MESSAGES
mkdir -p locale/hi/LC_MESSAGES

# Per-app locale dirs — run for each app that has _("...") strings
mkdir -p mysite/locale/en/LC_MESSAGES
mkdir -p mysite/locale/hi/LC_MESSAGES
```

## 4. Mark Strings in Code — What to Use Where

### Models (`models.py`)

```python
from django.utils.translation import gettext_lazy as _

class ActualSale(models.Model):
    period_type = models.CharField(
        _("period type"),           # ← field label in forms and admin
        max_length=16,
        choices=PERIOD_TYPE_CHOICES,
        help_text=_("Granularity of this period bucket."),
    )

    class Meta:
        verbose_name        = _("actual sale")
        verbose_name_plural = _("actual sales")
```

### Choice tuples in models

```python
# CORRECT — wrap the display label, not the value
PERIOD_TYPE_CHOICES = [
    ("month",    _("Month")),
    ("quarter",  _("Quarter")),
    ("year",     _("Year")),
]

# The stored DB value ("month", "quarter") is never translated —
# it's a code, not a user-facing string.
```

### Forms (`forms.py`)

```python
from django.utils.translation import gettext_lazy as _

class ActualSaleImportForm(forms.Form):
    period_type = forms.ChoiceField(
        label=_("Period Type"),
        choices=PERIOD_TYPE_CHOICES,
    )
    file = forms.FileField(
        label=_("Upload file"),
        help_text=_("Accepted formats: .csv, .xlsx"),
    )
```

### Validation errors in `clean()` methods

```python
from django.utils.translation import gettext_lazy as _

def clean(self):
    from django.core.exceptions import ValidationError
    if self.period_start and self.period_start.day != 1:
        raise ValidationError(
            _("Period start must be the first day of the month.")
        )
```

### Views (`views.py`)

Inside a view function/method you are already in a request context,
so you can use either — `gettext_lazy` is fine and consistent:

```python
from django.utils.translation import gettext_lazy as _
from django.contrib import messages

def upload_actuals_view(request):
    messages.success(request, _("Actuals uploaded successfully."))
    messages.error(request, _("Upload failed. Check the error log."))
```

### Templates

Templates use the `{% trans %}` tag — no Python import needed:

```html
{% load i18n %}

<h1>{% trans "Demand Planning" %}</h1>
<p>{% trans "Upload your actuals file below." %}</p>

{# For strings with variables: #}
{% blocktrans with count=object_list|length %}
    {{ count }} actuals records loaded.
{% endblocktrans %}
```

## 5. `makemessages` — Extracting Strings into `.po` Files

Run this from the project root **after** marking strings with `_("...")`:

```bash
# Extract strings from all apps into each app's locale/ dir
python manage.py makemessages --all --ignore=venv --ignore=node_modules

# Or for a specific language only:
python manage.py makemessages -l hi

# For JavaScript files (if you have JS that uses Django's i18n):
python manage.py makemessages -d djangojs -l hi
```

This scans every `.py` and `.html` file for `_("...")`, `{% trans %}`,
and `{% blocktrans %}` and writes/updates `.po` files.

**You do NOT hand-write `.po` files** — `makemessages` generates them.
You only fill in the `msgstr` values.

## 6. The `.po` File Format — What to Fill In

After `makemessages`, open e.g. `mysite/locale/hi/LC_MESSAGES/django.po`:

```po
# Hindi translations for mysite
# Generated by Django

msgid ""
msgstr ""
"Content-Type: text/plain; charset=UTF-8\n"
"Content-Transfer-Encoding: 8bit\n"
"Language: hi\n"

#: mysite/models/demand/actuals.py:45
msgid "period type"
msgstr "अवधि प्रकार"

#: mysite/models/demand/actuals.py:50
msgid "Month"
msgstr "माह"

#: mysite/models/demand/actuals.py:51
msgid "Quarter"
msgstr "तिमाही"

#: mysite/models/demand/actuals.py:52
msgid "Year"
msgstr "वर्ष"

#: mysite/models/demand/actuals.py:88
msgid "actual sale"
msgstr "वास्तविक बिक्री"

#: mysite/models/demand/actuals.py:89
msgid "actual sales"
msgstr "वास्तविक बिक्रियाँ"
```

Rules:
- `msgid` — the original English string (never change these)
- `msgstr` — the translation (fill these in)
- Leave `msgstr ""` for strings you haven't translated yet — Django falls back to `msgid`
- Lines starting with `#:` are source file references — auto-generated, don't edit

## 7. `compilemessages` — Compiling `.po` → `.mo`

Django does **not** read `.po` files at runtime. It reads compiled binary `.mo` files.
After filling in translations, compile:

```bash
python manage.py compilemessages
```

This produces `django.mo` alongside each `django.po`. The `.mo` file is what
Django actually loads. **Run `compilemessages` after every edit to a `.po` file.**

Add both to version control:

```
# .gitignore — do NOT ignore these:
# mysite/locale/  ← keep entirely
# locale/         ← keep entirely
```

## 8. Workflow Summary (Day-to-Day)

```
1. Write code with _("string") markers
         ↓
2. python manage.py makemessages --all
   (updates .po files, adds new msgid entries)
         ↓
3. Fill in msgstr values in .po files
   (use a tool like Poedit, or edit raw)
         ↓
4. python manage.py compilemessages
   (writes .mo files)
         ↓
5. Restart dev server (Django caches .mo at startup)
```

For production deploys, `compilemessages` must run as part of your deploy script,
before the app server starts. Add it to your `Makefile` or CI pipeline:

```bash
# In your deploy script, after collectstatic:
python manage.py compilemessages
```

## 9. Existing Codebase — Retroactive Addition

Since your existing models don't use `_("...")` yet, here's the safest way
to add it incrementally without touching migration state:

**`verbose_name` and `verbose_name_plural` in `Meta`:**
These are read at runtime (not stored in migrations). You can add `_("...")`
around them in any existing model with **no migration needed**.

```python
# Before
class ClientLocation(models.Model):
    class Meta:
        verbose_name = "client location"
        verbose_name_plural = "client locations"

# After — zero migration impact
class ClientLocation(models.Model):
    class Meta:
        verbose_name = _("client location")
        verbose_name_plural = _("client locations")
```

**Field `verbose_name` (first positional arg or kwarg):**
Also runtime-only. No migration needed.

```python
# Before
name = models.CharField("name", max_length=255)

# After — no migration
name = models.CharField(_("name"), max_length=255)
```

**`choices` display labels:**
The display label is runtime-only. The stored value (the code) is never affected.
No migration needed.

```python
# Before
STATUS_CHOICES = [("active", "Active"), ("inactive", "Inactive")]

# After — no migration
STATUS_CHOICES = [("active", _("Active")), ("inactive", _("Inactive"))]
```

**`help_text`:**
Runtime-only. No migration needed.

**`default` values:**
Never wrap a field `default` in `_("...")`. Defaults are stored in migrations
and evaluated at migration time — translation context doesn't exist there.

```python
# WRONG — breaks migrations
message = models.TextField(default=_("This feature is unavailable."))

# CORRECT — plain string for defaults
message = models.TextField(default="This feature is unavailable.")
```

**`validators` and `error_messages`:**
Use `_("...")` freely — these are runtime-only.

## 10. Interaction with `django-modeltranslation`

These two systems do not conflict. A single field can participate in both:

```python
# models.py
class Product(models.Model):
    name = models.CharField(_("name"), max_length=255)
    # django-modeltranslation adds: name_en, name_hi columns
    # _("name") controls the field label in admin/forms
    # name_en / name_hi store the actual translated content values

    class Meta:
        verbose_name = _("product")
```

`_("name")` → controls what the **label** says in the admin form header.
`name_en` / `name_hi` → controls what **content** is stored and served.

Both are needed in a fully localised app. The label "Name" appears in Hindi
as "नाम" (via `_("...")`), and the value stored in the name field is also
in Hindi (via `django-modeltranslation`).

## 12. `gettext_lazy` in `utils/feature_control.py`

The only translatable string in your utility is the fallback message.
Since this is a utility module (not a model), use `gettext_lazy` at
module level for the constant, and plain `gettext` inside functions
if you ever build dynamic messages:

```python
from django.utils.translation import gettext_lazy as _

# In celery_demand_feature_guard() return dict:
return {
    "status": "skipped",
    "feature": feature,
    "reason": result["message"] or _("This feature is currently disabled."),
}

# In demand_feature_required() decorator:
return render(
    request,
    template,
    {"message": result["message"] or _("This feature is currently unavailable.")},
    status=403,
)
```

## 13. Tools That Help

**Poedit** (https://poedit.net) — GUI editor for `.po` files. Makes translation
work much faster than editing raw text. Free version is sufficient.

**django-rosetta** — Web-based `.po` editor you can deploy alongside your app,
letting non-developers do translations in the browser without touching files.
Install: `pip install django-rosetta` and add to `INSTALLED_APPS`.

**VS Code extension** — "gettext" extension adds syntax highlighting for `.po` files.
